# Vibe Coding Skill — 阿蝶的 Vibe Coding Skill

## 概述
一个交互式的全流程开发引导 Skill。不只是文档，更是 Agent 和用户之间的开发向导。
在每个阶段告诉用户"你现在在哪、下一步做什么"，让开发过程有章可循。

## 触发条件
当用户意图开始一个新项目时自动触发。关键词包括但不限于：
"新建项目"、"开始新项目"、"初始化项目"、"从零开始开发"、"kickoff"、"vibe coding"、"开始开发"

## Skill 的作用
1. **启动引导**：用户开始新项目时，Agent 主动建议调用此 Skill，给出全流程概览
2. **步骤指引**：每到一个阶段，告诉用户当前该做什么、需要提供什么
3. **进度查询**：开发到任何阶段，用户可以问"我们现在到哪了"或"下一步干啥"，Agent 根据此 Skill 回答
4. **文档同步提醒**：需求变更时提醒用户同步更新文档
5. **经验沉淀**：开发过程中的经验教训自动往 AGENTS.md 归纳

---

## 全流程 MAP（快速总览）

```
┌─────────────────────────────────────────────────────┐
│  阶段0  口喷需求 → AI 生成结构化问卷                  │
│  阶段1  用户填写问卷 + 提供 GPT 对话链接 → 收集输入    │
│  阶段2  提炼需求决策摘要 → 生成三份核心文档             │
│        （PRD.md / UI.md / TECH.md）                  │
│  阶段3  生成 AGENTS.md + CLAUDE.md 软链接              │
│  阶段4  生成 MVP 开发计划（Plan V1）                   │
│  阶段5  执行 Plan → 开发 → 测试 → Debug               │
│  阶段6  MVP 成型 → 迭代优化 → 本地调试                 │
│  阶段7  部署上线                                      │
│  阶段8  持续维护：文档同步 + 经验沉淀                   │
└─────────────────────────────────────────────────────┘
```

**用户可以随时问 Agent："我们现在到哪一步了？" Agent 会读取此 Skill 并回答。**

## ⚠️ 全局规则（贯穿全程，优先级最高）

> 以下规则**不是某个阶段的事**，而是从项目开始到结束都必须遵守的铁律。

### 规则一：文档同步机制

**所有活文档（PRD.md、UI.md、TECH.md、PLAN.md）贯穿开发全程，始终与最新需求同步。**

需求变更时更新哪份文档：

| 变更类型 | 更新哪些文档 |
|---------|-------------|
| 产品功能变化 | PRD.md + PLAN.md |
| UI 风格/交互变化 | UI.md + PLAN.md |
| 技术架构/方案变化 | TECH.md + PLAN.md |
| MVP 范围变化 | PLAN.md + PRD.md |

> AGENTS.md **不会**因为每次产品/UI变化而更新。
> AGENTS.md 只记录：项目级约束、关键产品原则、工程纪律、通用经验教训。
> 只有当变更影响到"AI 每次启动都需要知道的底层规则"时，才更新 AGENTS.md。

**文档不允许长期落后于代码和用户最新需求。**
Agent 应在每次需求变更后主动提醒："需要同步更新 XX.md 吗？"

### 规则二：PLAN.md 是活的计划书

PLAN.md 不是写一次就完的文件，它是**贯穿项目全程的活计划书**。

**每条 Task 必须标注状态：**
- `⬜ 待执行` — 尚未开始
- `🔄 进行中` — 正在开发
- `✅ 已完成` — 测试通过
- `🚫 已废弃` — 需求变更后不再需要
- `⚠️ 阻塞` — 等待依赖或用户确认

**更新时机：**
- 每完成一个 Task → 更新状态为"已完成"
- 每轮需求变更 → 检查 PLAN.md，废弃过时 Task、新增新 Task
- 发现新问题或技术债 → 新增 Task
- Plan 调整后向用户展示变更摘要

### 规则三：Git 提交提醒

**每一轮修改完成后，Agent 必须在最后一句话提醒用户：**

> "这轮改动做完了，要不要 git commit 一下？"

用户可以选择提交或不提交，Agent 不强制，但**必须提醒**。
这条规则写入 AGENTS.md 的工程纪律中。

## 阶段0：口喷需求 & AI 生成问卷

### 触发方式
用户口喷需求想法（语音或文字均可），AI 根据内容生成结构化问卷。

### 问卷规范
- **问卷形式**：HTML 文件（美观、可离线打开、支持勾选和填写）
- **题目总数上限**：所有问卷合计不超过 200 题（太多了用户填不动）
- **每份问卷底部**：必须预留"手动补充区"，让用户自由补充题目没覆盖的内容
- **问卷分类**（根据项目实际需要灵活组合，不必每类都单独一份）：

| 问卷类型 | 覆盖内容 |
|---------|---------|
| 需求确认问卷 | 产品逻辑、核心功能、用户场景、业务流程、角色权限 |
| 核心模块问卷 | 特定重点模块的深度需求（如某个复杂功能的细节） |
| 前端UI问卷 | 页面结构、交互方式、组件偏好、响应式、动效 |
| 技术架构问卷 | 前后端架构、数据库、部署、安全、第三方集成 |
| PRD问卷 | 功能优先级、MVP范围、验收标准、用户旅程 |

> 以上分类可以合并（比如2-3份问卷涵盖全部），按项目复杂度决定。

### 同时收集
- 用户与 GPT（或其他 AI）的完整对话链接（不限轮数，整个对话都要）
- 对话中可能包含需求讨论、方案对比、用户补充说明等关键信息

## 阶段1：收集输入文件

### 输入文件清单
| 类型 | 说明 |
|------|------|
| 问卷文件（若干份） | 阶段0生成，用户填写完成后的 HTML 文件 |
| GPT 对话链接 | 用户提供的完整对话链接（Agent 通过链接读取全部内容） |

### Agent 处理流程
1. 逐份读取所有问卷 HTML，提取已选答案和补充内容
2. 读取 GPT 对话链接，提取全部对话内容
3. 交叉比对所有输入，提取完整需求点
4. 识别冲突、矛盾和遗漏 → 标记并**向用户确认**
5. 确认无误后进入文档生成阶段

### 材料冲突优先级（重要！）
当不同材料之间出现冲突时，按以下优先级判断（高优先级覆盖低优先级）：

| 优先级 | 来源 |
|--------|------|
| 1（最高） | 用户在 GPT 对话**后期**明确纠正过的内容（"不对"、"不是这样"、"我想要的是……"） |
| 2 | HTML 问卷中用户**手写补充框**的内容 |
| 3 | HTML 问卷中用户**勾选**的选项 |
| 4 | GPT 对话中的**早期**设想 |
| 5（最低） | Agent 自己的通用推断 |

> 如果无法判断，在文档中标记为 **【待用户确认】** 并列出具体问题。Agent 不得擅自决定。

## 阶段2：生成三份核心文档

### 2.0 先生成需求决策摘要（写 PRD 前必须做！）

> **在写正式 PRD 之前，Agent 必须先生成 FINAL_REQUIREMENTS_SUMMARY.md。**
> 这份摘要把原始问卷和 GPT 对话中的复杂信息压缩成**最终可执行结论**。

**摘要必须覆盖：**
- 产品定位、目标用户
- V1 范围、正式版范围
- 开发阶段划分（见下方"四阶段部署路径"）
- 核心模块和学习路径
- AI 调用方式（API Key 策略、缓存、Prompt 模板）
- 数据策略（云端/本地、同步/缓存）
- UI 风格、导航结构
- 技术栈选型
- 安全要求
- **明确不做的内容**

**不确定项用 【待用户确认】 标出。**

**用户审核摘要通过后，才开始写 PRD。这样避免返工。**

### 2.1 PRD.md — 产品需求文档

> 作用：让 AI 和人都清楚"做什么、怎么做、做到什么程度算完成"

包含：
- **产品概述**：一句话描述、目标用户、核心价值
- **用户画像**：客户群体、使用场景、痛点
- **功能清单**：按优先级 P0 / P1 / P2 排列
- **用户旅程**：User Story 格式，描述典型用户的使用路径
- **功能详情**：每个功能的详细描述 + 验收标准（Done Criteria）
- **信息架构**：页面层级关系、导航结构
- **MVP 范围**：V1 包含哪些、明确不包含哪些
- **非功能需求**：性能、兼容性、无障碍等

---

### 2.2 UI.md — 前端 UI 文档

> 作用：前端开发的唯一 UI 参考源

包含：
- **设计规范**：色彩、字体、间距、圆角等 Design Token
- **页面布局**：每个页面的结构描述
- **组件清单**：
  - 组件名称、用途、Props 接口
  - 组件状态（默认 / hover / active / disabled / error）
  - 组件代码示例或推荐组件库用法
- **响应式策略**：断点、适配方案
- **交互动效**：过渡、动画、反馈机制
- **路由结构**：页面间导航关系
- **UI 素材参考**：当用户需要 UI 灵感或组件实现参考时，Agent 应主动推荐前端 UI 素材库
  - 例如：[React Bits](https://reactbits.dev) — React 组件和动效灵感库
  - 例如：Shadcn/ui、Ant Design、Vuetify 等主流组件库
  - 用户问"这个怎么做"时，Agent 先查素材库再给方案

---

### 2.3 TECH.md — 技术文档

> 作用：技术实现的总纲，AI 开发时的技术决策依据

包含：
- **技术栈选型**：前端框架、后端框架、数据库、缓存等
- **系统架构**：前后端分层、服务间关系图
- **目录结构约定**
- **API 设计**：接口列表、请求/响应格式、错误码规范
- **数据库 Schema**
- **部署方案**：开发 / 测试 / 生产环境、CI/CD
- **第三方服务集成**：AI API、支付、短信等
- **功能实现方案**：PRD 中各功能对应的技术实现路径
- **前端实现方案**：状态管理、路由、数据请求

#### ⚠️ 安全专项（重点！）

> **API Key 安全是重中之重。绝不能让用户通过解包客户端拿到密钥。**

**安全分级设计（按阶段递进）：**

**V1 最低要求（开发阶段就做到）：**
- API Key 放在后端环境变量（`.env`，不入 Git）
- 前端只请求自己的后端，后端代理调用大模型 API
- API Key 绝不出现于：前端代码、APK、接口请求/响应、日志、Git 仓库

**正式版增强（上线前必须完成）：**
- 用户自带 API Key（BYOK）支持
- 用户 Key 传输必须加密（HTTPS + 额外加密层）
- 后端接收到用户 Key 后加密存储（不明文落库）
- 前端展示 Key 时做脱敏处理（如 `sk-****abcd`）
- 支持更换/删除 Key
- 调用频率限制（Rate Limit）+ 异常监控 + 自动熔断
- 定期轮换预置 Key，单个 Key 泄露不影响全局
- CI/CD 加入 Secret 扫描（如 gitleaks）
- 日志脱敏（Key 部分自动替换为 `***`）

## 阶段3：生成 AGENTS.md & CLAUDE.md

> **此阶段在文档生成后、开发开始前完成。是开发的前置条件。**

### AGENTS.md — 项目级 Agent 持久化指令

AI 每次启动时自动读取，包含 AI 无法自行推断的业务上下文。

**必须包含：**
- **项目背景**：产品是什么、场景描述
- **目标用户群体画像**
- **产品原型概述**（当前阶段长什么样）
- **关键产品原则**（AI 最容易搞错的事，比如"不是传统背单词APP"、"以词族为核心"等）
- **项目约定**（命名规范、分支策略、提交规范等）
- **关键技术决策及原因**
- **不可违背的约束条件**（如安全红线、UI风格约束）
- **工程纪律**（必须写入）：
  - 开发前先读 PRD.md、UI.md、TECH.md、AGENTS.md
  - 不得擅自扩展需求
  - 不得恢复已废弃方案
  - 不得把 API Key 写进前端
  - 修改功能时同步更新文档
  - 文档和用户最新语言冲突时，以用户为准，并回写文档
  - **每轮修改完成后，必须提醒用户是否要 git commit**
- **常见踩坑记录**（开发中持续沉淀）

**维护规则：**
- Agent 开发过程中往 memory 积累会话经验
- 定期 review，将通用经验提炼进 AGENTS.md
- 过时条目主动清理；被用户推翻的内容删除或标记废弃
- AGENTS.md 是精炼的项目知识，不是日志，应保持**短、准、稳定**

### CLAUDE.md — 软链接

```
# 只维护 AGENTS.md 一份，CLAUDE.md 做软链接
# Windows:   mklink CLAUDE.md AGENTS.md
# Linux/Mac: ln -s AGENTS.md CLAUDE.md
```

- Claude Code 读 CLAUDE.md（软链接 → AGENTS.md）
- 其他 Agent（Cursor、Codex等）直接读 AGENTS.md
- **每个项目都必须做这个软链接**

## 阶段4：生成 MVP 开发计划

### 做什么
AI 基于 PRD.md + UI.md + TECH.md + AGENTS.md，生成可执行的开发计划。

### Plan V1 要求
- 拆解为具体 Task，粒度合理
- 按依赖关系排序
- 每个 Task 包含：任务描述、涉及文件、预期产出、验收方式
- 标记 MVP 必须 vs 后续迭代
- 每条 Task 初始状态为 `⬜ 待执行`
- 预估复杂度

### 输出
- PLAN.md（或 plan/ 目录）
- **用户审阅确认后**才进入执行阶段

> **PLAN.md 是活计划书**，贯穿项目全程。详见全局规则二。

---

## 阶段5：执行计划 & 测试 Debug

1. Agent 按 Plan 逐个 Task 执行
2. 每完成一个 Task → 运行测试 → 确认通过
3. 遇到 Bug → 修复 → 回归测试
4. Task 间保持代码可运行，不积攒到最后一块测

**测试策略：**
- 单元测试（关键函数/组件）
- 集成测试（API 接口、组件交互）
- 手动验收（对照 PRD 验收标准）

---

## 阶段6：迭代优化 & 本地调试

- 根据 PRD 中 P1/P2 功能逐批添加
- UI 打磨（动画、响应式、细节交互）
- 性能优化
- 每轮迭代本地调试通过
- **每次变更同步更新文档**

---

## 阶段7：部署上线

### 四阶段部署路径（推荐）

> 从本地到线上，分四步走，不跳步，每步验证通过再进下一步。

| 阶段 | 做什么 | 验证什么 |
|------|--------|---------|
| **第一步：本地跑通** | 前后端+数据库全在本机运行，跑通核心链路 | 功能可用，不被服务器/域名/备案拖慢 |
| **第二步：局域网真机测试** | 手机连接电脑局域网 IP，真机操作 | 手机体验、点击热区、列表性能、弹层速度 |
| **第三步：云服务器内测** | 后端+数据库部署到云服务器（阿里云/腾讯云等），连接公网 IP | 多人使用、网络延迟、数据持久化 |
| **第四步：正式版上线** | 绑定域名 + HTTPS + 备案 + 上架 | 安全合规、BYOK、多设备同步、完整功能 |

> 每个阶段的 API_BASE_URL 不同，TECH.md 中必须集中配置，不要散落在页面代码里。
> 示例：`localhost:8000` → `192.168.x.x:8000` → `公网IP:8000` → `https://api.xxx.com`

### 部署前检查清单
- [ ] 所有 P0 功能测试通过
- [ ] 本地调试无明显 Bug
- [ ] 安全检查通过（API Key 无泄露、CORS 正确、输入校验完备）
- [ ] 环境变量配置完成
- [ ] TECH.md 部署方案已就绪

### 执行
- 按 TECH.md 部署方案执行
- 部署后冒烟测试
- 记录部署信息

---

## 阶段8：持续维护

- 需求变更 → 按全局规则同步更新文档
- AGENTS.md 定期 review 和提炼（只沉淀通用经验，不记录每次变更）
- memory 中经验定期整理进 AGENTS.md
- 过时内容主动清理

| `TECH.md` | 技术文档 |

> 但不要删除，它是需求溯源的依据。

## 交互指令（用户随时可用）

| 用户说 | Agent 做什么 |
|--------|-------------|
| "我们现在到哪了" / "下一步干啥" | 读取当前进度，回答当前阶段和下一步 |
| "跳到阶段X" | 跳转到指定阶段，检查前置条件是否满足 |
| "更新需求：XXX" | 修改需求，同步更新 PRD/UI/TECH 三份文档 |
| "review 一下文档" | 检查三份文档是否与最新需求一致 |
| "安全检查" | 重点检查 API Key、认证、加密等安全相关内容 |
| "生成部署清单" | 生成阶段7的部署前检查清单 |

## 关键原则

1. **文档先行**：没有文档不写代码，文档是需求的唯一真相源
2. **禁止直接开发**：Agent 完成 FINAL_REQUIREMENTS_SUMMARY + PRD + UI + TECH + AGENTS + PLAN 之前，不得写业务代码
3. **文档是活的**：需求变了文档必须跟着变
4. **安全第一**：API Key 绝不暴露给客户端，这是红线
5. **AGENTS.md 是项目大脑**：沉淀 AI 猜不到的业务上下文
